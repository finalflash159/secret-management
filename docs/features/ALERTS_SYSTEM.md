# Alert System

## Overview

The Alert System provides notifications to users about important events, security issues, and system updates. Alerts help keep users informed about changes within their organizations and projects.

## Alert Types

| Type | Category | Description |
|------|----------|-------------|
| `info` | General | General information messages |
| `warning` | General | Warning messages |
| `error` | General | Error messages |
| `success` | General | Success messages |
| `secret_expiry` | Secrets | Secret expiring soon |
| `secret_expired` | Secrets | Secret has expired |
| `security` | Security | Security alerts |
| `member_added` | Members | New member added |
| `member_removed` | Members | Member removed |
| `access_granted` | Access | Access was granted |
| `access_revoked` | Access | Access was revoked |

## Alert Structure

### Alert Fields

```typescript
interface Alert {
  id: string;
  userId: string;           // Owner of the alert
  orgId?: string;           // Related organization
  projectId?: string;       // Related project
  type: AlertType;          // Alert category
  title: string;            // Short title
  message: string;          // Detailed message
  link?: string;            // Navigation link
  read: boolean;           // Read status
  readAt?: DateTime;        // When marked as read
  metadata?: object;        // Additional data
  createdAt: DateTime;      // Creation timestamp
}
```

## Alert Flow

### Creating Alerts

Alerts are created automatically by the system when events occur:

```
Event Occurs
    │
    ▼
System Creates Alert
    │
    ├── Secret expiring in 7 days
    ├── Member added to org
    ├── Secret expired
    └── Security issue detected
    │
    ▼
User Receives Alert
    │
    ├── Notification in header (bell icon)
    └── Alert appears in /alerts page
```

### Alert Lifecycle

1. **Created**: Alert is generated by system
2. **Unread**: User hasn't seen the alert
3. **Read**: User clicks/views the alert
4. **Deleted**: User dismisses or auto-cleanup removes it

## User Interface

### Header Notification

The header includes a bell icon showing:

- Number of unread alerts
- Quick link to alerts page
- Click to view dropdown with recent alerts

### Alerts Page

The global alerts page (`/alerts`) shows:

- All alerts across all organizations
- Filter by organization, project, type
- Filter by read/unread status
- Mark as read
- Delete alerts

### Alert Card

Each alert displays:

- **Icon**: Based on alert type
- **Title**: Brief description
- **Message**: Full details
- **Source**: Organization/project name
- **Time**: Relative time (e.g., "2 hours ago")
- **Actions**: Mark as read, Delete

---

## Managing Alerts

### Mark as Read

Individual alerts can be marked as read:

1. Click on the alert
2. Navigate via the link
3. Alert automatically marked as read

### Delete Alerts

Single alert or bulk delete:

```bash
# Delete single alert
DELETE /api/alerts?id=alert-id

# Cleanup old read alerts
DELETE /api/alerts
```

### Auto-Cleanup

Read alerts older than 30 days are automatically cleaned up when calling the cleanup endpoint.

## Technical Implementation

### Database Schema

```prisma
model Alert {
  id          String    @id @default(cuid())
  userId      String
  user        User      @relation(fields: [userId], references: [id])
  orgId       String?
  org         Organization? @relation(...)
  projectId   String?
  project     Project?  @relation(...)
  type        AlertType
  title       String
  message     String
  link        String?
  read        Boolean   @default(false)
  readAt      DateTime?
  metadata    Json?
  createdAt   DateTime  @default(now())

  @@index([userId, read, createdAt])
}

enum AlertType {
  info
  warning
  error
  success
  secret_expiry
  secret_expired
  security
  member_added
  member_removed
  access_granted
  access_revoked
}
```

### API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | /api/alerts | List alerts |
| DELETE | /api/alerts?id={id} | Delete single alert |

## Best Practices

### Alert Design

1. **Be concise**: Short, descriptive titles
2. **Provide context**: Include relevant details in message
3. **Add links**: Help users navigate to relevant content
4. **Use appropriate types**: Match alert type to content

### User Experience

1. **Don't over-notify**: Avoid alert fatigue
2. **Group similar alerts**: Batch when possible
3. **Set priorities**: Important alerts more prominent
4. **Auto-cleanup**: Remove old alerts regularly

### Security

1. **Monitor security alerts**: Review immediately
2. **Set up notifications**: Email digests (future)
3. **Track patterns**: Identify recurring issues
4. **Document alerts**: Help users understand
